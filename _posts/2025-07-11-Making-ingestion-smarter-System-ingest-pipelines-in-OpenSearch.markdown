OpenSearch 3.1 introduces the system ingest pipeline, which enables plugin developers to automatically process documents during ingestion. This feature simplifies the document processing workflow by allowing developers to define custom system ingest processors that run based on fields or parameters specified in the index mapping.

Previously, plugin developers needed to create custom ingest processors and rely on users to manually configure ingest pipelines that explicitly invoked those processors. With the system ingest pipeline, this additional setup is no longer required. Instead, plugin developers can deliver a streamlined, automatic ingestion experience that reduces complexity for end users.

## System ingest pipeline vs. standard ingest pipeline

In OpenSearch, a standard ingest pipeline is defined using the [ingest pipeline API](https://docs.opensearch.org/docs/latest/ingest-pipelines/create-ingest/). End users must provide the configuration for each ingest processor they want to include in the pipeline. To apply a standard ingest pipeline, users must either specify it in the index settings or pass it as part of the indexing request.

A system ingest pipeline works similarly by using one or more ingest processors to transform documents during ingestion. However, the key difference is that system ingest pipelines are not defined by end users. Instead, they are automatically generated based on the index mapping and the system ingest processor factories provided by plugin developers.



| Pipeline type           | How to define it                                                                                           | How to trigger it                                                                                                            | How to disable it                                                                                       |
|-------------------------|------------------------------------------------------------------------------------------------------------|-----------------------------------------------------------------------------------------------------------------------------|----------------------------------------------------------------------------------------------------------|
| Normal ingest pipeline  | Users define standard ingest pipelines manually using the ingest pipeline API                              | Specified in the index settings as the default or final pipeline, or when passed as a parameter in an indexing request.     | Simply avoid referencing it in index settings or indexing requests.                                     |
| System ingest pipeline  | Automatically generated by OpenSearch based on index mappings and processor factories registered by plugin developers | Triggered automatically when the index mapping includes specific fields or parameters defined by plugin developers | Set the `cluster.ingest.system_pipeline_enabled` cluster setting to `false`.                            |

## How it works

When OpenSearch receives a document indexing request, it determines which ingest pipelines to apply. Starting in OpenSearch 3.1, this process includes resolving a system ingest pipeline based on the index mapping of the target index.

To enable system ingest pipelines, plugin developers register a system ingest processor and its associated factory. The factory evaluates the index mapping and decides whether to create a processor. If applicable, the processor is added to the automatically generated system ingest pipeline.

### Pipeline execution and update behavior
To optimize performance, OpenSearch caches the generated system ingest pipeline for each index. This cache is automatically invalidated when the index mapping changes, ensuring that pipelines remain consistent with the latest mapping.

During indexing, OpenSearch executes all applicable pipelines in order. The system ingest pipeline always runs after the default and final pipelines.

### Ingest Pipelines and Update Operations
In OpenSearch, users are able to update existing documents within their indices using [single update](https://docs.opensearch.org/docs/latest/api-reference/document-apis/update-document/)
and bulk operations. To do this, users include a "partial doc" in the update request containing the values they want to be updated, and the existing document is updating accordingly. Update operations have special interactions with ingest pipelines:

* Single update operations trigger default and final pipelines in all cases
* Bulk updates do not trigger any ingest pipelines on the "partial doc" included in the request unless doc_as_upsert flag is enabled, then the "partial doc" will trigger default and final pipelines. If the user includes an upsert document, ingest pipelines will be executed on that document as well

Making this behavior more configurable is under discussion in [this proposal](https://github.com/opensearch-project/OpenSearch/issues/17742).

Unlike default and final pipelines, system ingest pipelines are applied to all update operations (both single and bulk), ensuring system processors can maintain their fields even when updates occur after initial indexing. For example, the system_ingest_processor_semantic_field processor (introduced in Neural Search 3.1) uses system pipelines to  generate text embeddings at ingest time and regenerating them whenever their source text fields are updated.

Plugin developers should note that this means all system ingest processors must be able to handle the "partial doc" update case gracefully and cover the update scenario in their unit and integration tests, or else the processors may fail.

## Building a custom system ingest processor

Plugin developers can extend the system ingest pipeline by creating and registering a custom system ingest processor. To build a custom system ingest processor, follow these steps:

1. Implement a system ingest processor
   Create a class that extends the SystemIngestProcessor interface. Define the logic for transforming incoming documents.
2. Create a processor factory
   Implement a factory that extends SystemIngestProcessor.Factory. This factory determines whether the processor should be applied based on the index mapping.
3. Register the processor factory
   Register the factory in your plugin by overriding the getSystemIngestProcessorFactories() method.

Below are the detailed steps to create and register a system ingest processor in a plugin. You can also refer to this example for a complete implementation.

### step 1 Create an System Ingest Processor

```java
/**
 * This system ingest processor will add a field to the ingest doc.
 */
public class ExampleSystemIngestProcessor extends AbstractBatchingSystemProcessor {

    /**
     * The type of the processor.
     */
    public static final String TYPE = "example_system_ingest_processor";

    /**
     * Constructs a new ExampleSystemIngestProcessor
     * @param tag tag of the processor
     * @param description description of the processor
     * @param batchSize batch size which is used to control each batch how many docs the processor can process
     */
    protected ExampleSystemIngestProcessor(String tag, String description, int batchSize) {
        super(tag, description, batchSize);
    }

    @Override
    protected void subBatchExecute(List<IngestDocumentWrapper> ingestDocumentWrappers, Consumer<List<IngestDocumentWrapper>> handler) {
        // Add your logic to batch process the docs
        handler.accept(ingestDocumentWrappers);
    }

    @Override
    public void execute(IngestDocument ingestDocument, BiConsumer<IngestDocument, Exception> handler) {
        super.execute(ingestDocument, handler);
    }

    @Override
    public IngestDocument execute(IngestDocument ingestDocument) {
        // Add your logic to process the doc
        return ingestDocument;
    }

    @Override
    public String getType() {
        return TYPE;
    }
}
```

### step 2 Create an System Ingest Processor Factory

```java
/**
 * A factory to create the example system ingest processor
 */
public class ExampleSystemIngestProcessorFactory extends AbstractBatchingSystemProcessor.Factory {
    /**
     * The type of the factory.
     */
    public static final String TYPE = "example_system_ingest_processor_factory";
    /**
     * A default batch size.
     */
    private static final int DEFAULT_BATCH_SIZE = 10;

    /**
     * Constructs a new ExampleSystemIngestProcessorFactory
     */
    protected ExampleSystemIngestProcessorFactory() {
        super(TYPE);
    }

    @SuppressWarnings("unchecked")
    @Override
    protected AbstractBatchingSystemProcessor newProcessor(String tag, String description, Map<String, Object> config) {
        // Currently we only support index mapping data but in future we can support
        // using more info to control your system ingest processor
        final List<Map<String, Object>> mappings = new ArrayList<>();
        final Object mappingFromIndex = config.get(INDEX_MAPPINGS);
        final Object mappingFromTemplates = config.get(INDEX_TEMPLATE_MAPPINGS);
        if (mappingFromTemplates instanceof List) {
            mappings.addAll((List<Map<String, Object>>) mappingFromTemplates);
        }
        if (mappingFromIndex instanceof Map) {
            mappings.add((Map<String, Object>) mappingFromIndex);
        }

        // If no config we are not able to create a processor so simply return a null to show no processor created
        if (mappings.isEmpty()) {
            return null;
        }
        
        // Add your logic to decide if we should create the system processor
        // and how to create it
        return new ExampleSystemIngestProcessor(tag, description, DEFAULT_BATCH_SIZE)
    }
}
```

### step 3 Register the System Ingest Processor

```java
/**
 * Example plugin that implements a custom system ingest processor.
 */
public class ExampleSystemIngestProcessorPlugin extends Plugin implements IngestPlugin {
   /**
    * Constructs a new ExampleSystemIngestProcessorPlugin
    */
   public ExampleSystemIngestProcessorPlugin() {}

   @Override
   public Map<String, Processor.Factory> getSystemIngestProcessors(Processor.Parameters parameters) {
      return Map.of(ExampleSystemIngestProcessorFactory.TYPE, new ExampleSystemIngestProcessorFactory());
   }
}
```

## Summary

The system ingest pipeline in OpenSearch 3.1 offers plugin developers a powerful way to simplify document processing during ingestion. By automatically generating pipelines based on index mappings and registered processor factories, this feature removes the need for manual user configuration and ensures consistent, automatic execution of custom logic—such as generating embeddings with the Neural Search plugin. Unlike standard pipelines, system pipelines apply even during partial updates, making them especially useful for machine learning and search applications. As the feature evolves, enhancements like greater configurability are under active discussion, giving developers more flexibility in how and when their processors run.


## What’s next

To build on the foundation introduced in OpenSearch 3.1, upcoming enhancements to the system ingest pipeline aim to provide greater flexibility and developer tooling:

* [Configurable execution conditions](https://github.com/opensearch-project/OpenSearch/issues/17742): Allow users to control when the system ingest pipeline should run—for example, only on create, update, or full document replacements.
* [Simulation support](https://github.com/opensearch-project/OpenSearch/issues/18252): Enable users to simulate system ingest pipeline execution on sample documents, making it easier to validate processor behavior and debug issues during development.
* [Explain API for system ingest pipelines](https://github.com/opensearch-project/OpenSearch/issues/18253): Provide a way to view the generated pipeline for an index, showing which processors are included and their originating plugin or factory. This helps clarify how documents are being transformed during ingestion.
